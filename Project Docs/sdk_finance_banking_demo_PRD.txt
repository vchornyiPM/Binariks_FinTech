1. Purpose & Context
This document defines the requirements for a React Native mobile banking demonstration application built on the SDK.Finance REST API. The app is designed as a sales tool for Binariks account managers to show prospective clients how quickly a fully functional mobile banking product can be constructed using the SDK.Finance platform.

The application will be distributed via Expo Go (no App Store / Play Store deployment needed) and run on iOS and Android devices during live sales presentations.

1.1  Target Audience
• Primary users — Binariks Sales Managers (demoing the product)
• Secondary audience — Prospective clients watching the demo on a device
• Developer audience — Binariks engineers building the app via Claude Code

1.2  Business Goals
• Reduce time-to-demo from weeks to days
• Demonstrate SDK.Finance platform capabilities with real API calls
• Show breadth of mobile banking features (wallets, transfers, FX, history)
• Create a reusable, easily rebrandable template for client POCs

? This PRD is optimised for use with Claude Code. All API details, component specifications, and acceptance criteria are written so they can be handed directly to an AI-assisted developer session.

4. SDK.Finance API — Endpoint Mapping
All endpoints below are on base URL https://api-back.sdk.finance/api. The demo uses only customer-facing APIs (no admin endpoints). Authentication header required on all calls except /authorization and /registration.

  { "login": "user@email.com", "password": "SecurePass1!" }
Response — save token field from JSON response and inject as 'Authorization: Bearer {token}' header.

4.2  Wallets / Accounts (Coins)

  coinSerial, balance, currency.code, currency.symbol, name, type (INDIVIDUAL | MERCHANT)

4.3  Transfers (P2P / In-System)

3. POST /templates/transfer/{id}/calculate ? show fee to user

4.8  QR Payments
QR flow is partially simulated for demo purposes. The QR code encodes a coinSerial + amount. On scan:
6. Parse QR payload: { coinSerial, amount, currency, recipientName }
7. Pre-fill Send Money form
8. Execute via /templates/transfer/{id}/execute

? SDK.Finance supports PoS and QR natively in production. For the demo, generate QR codes client-side encoding coinSerial values from the sandbox environment.


5. Screen Specifications
5.1  Login Screen
• Email + Password fields with validation
• 'Sign In' button ? POST /authorization ? store JWT in expo-secure-store
• Auto-fill with demo credentials from .env for easy demo reset
• Loading spinner during auth ? navigate to Dashboard on success
• Error toast on 401 (wrong credentials)

? Demo tip: pre-populate fields on mount from DEMO_EMAIL / DEMO_PASSWORD env vars so the presenter just taps Sign In.

5.2  Dashboard Screen (Home Tab)
• Horizontal scrollable wallet cards showing each coin: balance, currency name, coin serial
• Total balance aggregated in a selected base currency (USD default)
• Quick action buttons: Send, Receive (QR), Exchange, Top-Up
• Recent transactions list (last 5) — tap ? Transaction Detail
• Pull-to-refresh refreshes GET /coins and POST /transactions/view

? UI priority: make the multi-currency balance cards visually stunning — this is the main 'wow' moment in the demo.

5.3  Send Money Screen
• Recipient search field: autocomplete against POST /users/view
• Select source wallet (coinSerial) from dropdown
• Amount input with currency symbol
• Fee preview: POST /templates/transfer/{id}/calculate shown inline
• Confirm button with summary sheet (recipient, amount, fee, total)
• Success animation + transaction ID display

5.4  Transaction History Screen (History Tab)
• Flat list with pull-to-refresh, infinite scroll (pageSize: 20)
• Filter bar: date range picker, type chip filter (All / Transfer / Exchange / Top-Up)
• Each row: icon by type, counterparty name, date, amount with +/- colour
• Tap row ? Transaction Detail sheet with full info
• Mini bar chart at top: 7-day spending by day (Victory Native)

5.5  Currency Exchange Screen (FX Tab)
• From / To currency selectors with flag emoji
• Amount input ? real-time preview via POST /exchange/calculate (debounced 500 ms)
• Live rate banner updated every 30 s via polling POST /exchange-rates/view
• 'Exchange Now' ? POST /exchange/execute ? success screen
• Rate history mini sparkline (last 10 polling results, in-memory)

5.6  QR Payment Screen
• Two tabs: Scan (camera) and Show My QR
• 'Show My QR': renders QR encoding { coinSerial, name } for the current default wallet
• 'Scan': expo-barcode-scanner ? parse payload ? navigate to pre-filled Send Money
• Supports simulated QR codes printed on demo materials

5.7  Profile / KYC Screen (Profile Tab)
• User avatar (initials fallback), name, email
• KYC Status badge: Verified / Pending / Not Started (from GET /kyc/status)
• Account info: registration date, user ID
• App version, Demo Mode indicator
• Logout button ? clear secure store ? navigate to Login

6.1  Auth Gate
Root _layout.tsx checks expo-secure-store for a valid token on mount. If missing ? redirect to (auth)/login. On token expiry (401 response interceptor in Axios) ? clear store ? redirect to login.


7. State Management
Use Zustand stores for global state. Keep API logic in service files, not components.

StoreState FieldsActionsauthStoretoken, user, isAuthenticatedlogin(), logout(), setUser()walletStorecoins[], selectedCoin, loadingfetchCoins(), setSelectedCoin()txStoretransactions[], filters, page, totalfetchTransactions(), setFilters(), loadMore()fxStorerates{}, lastUpdated, loadingfetchRates(), calculate()uiStoreisLoading, toast{}, activeTabshowToast(), hideToast()
7.1  API Service Layer
Create /services/api.ts as the Axios instance with base URL and auth interceptor:
// services/api.ts const api = axios.create({ baseURL: process.env.EXPO_PUBLIC_API_URL }); api.interceptors.request.use(cfg => {   const token = secureStore.getItemAsync('jwt');   cfg.headers.Authorization = `Bearer ${token}`;   return cfg; }); api.interceptors.response.use(null, err => {   if (err.response?.status === 401) authStore.logout();   return Promise.reject(err); });


8. UI / UX Design Guidelines
8.1  Design Tokens
TokenValueUsagePrimary Blue#1A56DBButtons, active tabs, highlightsDark Navy#0E4D99Headings, wallet card gradientsBackground#F5F7FAScreen backgroundCard BG#FFFFFFCard and list item backgroundsSuccess Green#10823BPositive amounts, verified badgeError Red#9B1C1CNegative amounts, errorsText Primary#1A1A2EMain textText Secondary#6B7280Labels, metadataBorder#E5E7EBDividers, input borders
8.2  Component Guidelines
• WalletCard — Gradient card (navy?blue) with coin balance. Use LinearGradient from expo-linear-gradient
• TransactionRow — Left icon (color by type), centre text (name + date), right amount (green/red)
• CurrencyInput — Large numeric input with currency badge, locale formatting (toLocaleString)
• StatusBadge — Rounded pill: VERIFIED (green), PENDING (amber), FAILED (red)
• ConfirmSheet — Bottom sheet (react-native-bottom-sheet) for transfer/exchange confirmations
• SkeletonLoader — Placeholder shimmer while data loads — never show empty screens

8.3  Demo Experience Rules
• All network errors must show a user-friendly toast — never raw error objects
• Optimistic UI: update balance display immediately on transfer, reconcile on refresh
• Smooth transitions: use react-native-reanimated for card flip, number counting animations
• Demo Reset: Settings screen (hidden, 5-tap on logo) that clears all state and logs out
• Offline banner: show 'Demo Mode — No Connection' if network unavailable

13. Known Limitations & Out of Scope
13.1  Demo Limitations
• Sandbox data only — no real money movement
• QR payments are simulated (no physical PoS terminal integration)
• Push notifications not implemented in Expo Go (would require EAS Build)
• Biometric auth (FaceID / TouchID) not in scope for v1
• Multi-language (i18n) not implemented — English only

13.2  Out of Scope (v1)
• Loan / credit module
• PDF report generation
• Admin / back-office functionality
• PayPal / Stripe integration
• SWIFT / SEPA wire transfers UI
• Merchant PoS dashboard
• Fraud detection dashboard

13.3  Production Readiness Path
To convert this demo into a production app, the following additional steps are required:
9. Switch from Expo Go to EAS Build (managed or bare workflow)
10. Add real SDK.Finance production credentials and bank integration
11. Implement push notifications (Expo Notifications + webhook receiver)
12. Add biometric authentication (expo-local-authentication)
13. Implement proper error monitoring (Sentry)
14. Add App Store / Play Store submission pipeline
15. Security audit and penetration testing
